# Claude API Messages 是什么

## 这是什么

`Claude API Messages` 是你和 Claude 模型进行一次次对话请求的核心接口。你把消息历史、系统指令、模型名、最大输出长度等信息发给它，它返回 Claude 的回答。

可以先把它理解成：

- 这是 Claude API 里最基础的“对话入口”
- 很多更复杂的能力，本质上都建立在它之上
- 你后面看到的工具调用、视觉输入、多轮对话，很多都还是围绕这个接口展开

## 它解决什么问题

它解决的是“我的程序要怎样把问题发给 Claude，并拿回结构化响应”。

如果不用这个接口，你就没法在自己的应用里：

- 发起普通文本对话
- 维护多轮上下文
- 让 Claude 决定是否调用工具
- 读取停止原因，决定程序下一步该怎么走

所以它不是“某个高级功能”，而是整个 Claude 应用开发里的基础层。

## 一个最小心智模型

你可以把一次调用想成下面这件事：

1. 你把当前完整上下文打包
2. 发送给 `POST /v1/messages`
3. Claude 生成回复
4. 你根据返回结果继续显示、追问，或者执行工具

也就是说，`Messages API` 更像“无状态请求接口”，而不是一个自动帮你记住所有历史的聊天会话服务。

## 为什么它很重要

后面很多概念都要依赖它：

- `tool_use` 要靠它返回工具调用意图
- `stop_reason` 要靠它告诉你这轮为什么停下
- 多轮对话要靠你自己把历史重新发回去
- 上下文管理、压缩、裁剪，都是围绕它的输入预算展开

如果这里没吃透，后面学 Agent SDK 或 MCP 时很容易“知道名词，但不知道底层怎么运作”。

## 请求长什么样

最常见的请求里，至少会看到这些字段：

```json
{
  "model": "claude-opus-4-6",
  "max_tokens": 1024,
  "system": "你是一个有帮助的助手。",
  "messages": [
    {
      "role": "user",
      "content": "Hello, Claude"
    }
  ]
}
```

先记住这几个字段的职责：

- `model`：选哪个模型
- `max_tokens`：这次最多允许 Claude 输出多少 token
- `system`：给模型设定长期行为规则
- `messages`：本轮可见的对话历史

## 核心概念

### 1. 它是无状态的

官方文档明确说 `Messages API` 是 `stateless`。意思不是“它不能多轮对话”，而是“它不会替你自动保存对话状态”。

所以每次请求时，你都要自己发送需要保留的历史消息。

这点非常关键，因为它直接决定：

- 为什么你要管理上下文窗口
- 为什么长对话会越来越贵
- 为什么摘要和裁剪是重要能力

### 2. `messages` 里放的是对话历史

常见角色包括：

- `user`
- `assistant`

如果是工具调用流程，还会出现工具相关的内容块。

学习时先不要把注意力过度放在“字段背诵”上，先抓住本质：

- Claude 只能看到你这次传进去的东西
- 你不传，它就不知道

### 3. `system` 不是普通聊天消息

`system` 用来规定角色、边界、输出方式。它和普通 `messages` 不一样，通常承担“全局约束”的职责。

比如：

- 你是客服助手
- 回答要简洁
- 输出必须遵守某个格式

### 4. 返回结果里要看 `stop_reason`

`stop_reason` 表示 Claude 为什么停下来。这个字段非常重要，因为程序下一步怎么做，常常由它决定。

常见值先记这几个：

- `end_turn`：Claude 正常说完了
- `tool_use`：Claude 想调用工具
- `max_tokens`：输出被截断了
- `stop_sequence`：遇到你定义的停止序列

对于做应用的人来说，`stop_reason` 不是“附带信息”，而是控制流程的重要信号。

## 一个最小示例

如果只是普通问答，程序流程很简单：

1. 把用户消息放进 `messages`
2. 调用 `POST /v1/messages`
3. 取出返回的文本内容
4. 检查 `stop_reason`

如果是多轮对话：

1. 保存前面用户和 assistant 的消息
2. 把完整历史再次发送
3. 让 Claude 在已有上下文上继续回答

## 容易错的点

### 1. 以为 API 会自动记住历史

这是最常见误解。`Messages API` 本身不会替你跨请求保存状态。多轮对话能成立，是因为你每次都把历史重新带回去了。

### 2. 只看文本，不看 `stop_reason`

如果你只关心返回了什么文字，而不关心为什么停下，就会把“该调用工具”“被截断了”“正常结束了”这些不同情况混在一起。

### 3. 工具结果后乱加说明文字

官方关于 `stop_reason` 的文档特别提醒，工具调用后如果你在 `tool_result` 后面又额外加文字，可能会让 Claude 学到错误模式，甚至出现空响应。

### 4. 还把预填充当成主流技巧

官方文档现在已经明确说明：把 assistant 的开头内容预先塞进最后一条消息里这种 `prefill` 技巧，已经不适用于部分较新的模型。更推荐结构化输出或系统提示。

## 站在考试角度，可能怎么考

这类题通常不会只问你“字段定义”，更可能考你是否理解流程。

常见考法会是：

- 为什么多轮对话要传完整历史
- 收到 `tool_use` 后程序该怎么做
- 收到 `max_tokens` 时意味着什么
- 为什么系统提示会影响工具选择
- 为什么上下文会因为工具结果不断膨胀

也就是说，考试更偏“系统行为理解”，不是背 API 文档目录。

## 你现在可以先记住的一句话

`Claude API Messages` 就是 Claude 应用开发的基础对话接口；它本身是无状态的，所以你必须自己管理历史、上下文和后续控制流程。

## 下一篇建议

学完这篇，最自然的下一篇就是：

- `Tool Use 是什么`

因为 `Messages API` 负责“对话入口”，而 `Tool Use` 负责“模型想调用外部能力时怎么表达”。
